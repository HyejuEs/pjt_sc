# 문서 파싱 검증 파이프라인 (WER + TEDS-content)

원본 문서(GT)를 파서가 얼마나 정확히 텍스트로 옮겼는지를 두 지표로 측정한다.
가이드라인 적용 **전(before)**/**후(after)** 두 파서 결과를 같은 GT와 비교해
개선폭까지 계산한다.

- **WER** (본문 텍스트, 낮을수록 좋음): 원문의 단어가 얼마나 그대로 보존됐는지.
  텍스트 지표는 WER 하나만 쓴다. 상황별 대응(띄어쓰기 밀림, 서식 채점 여부 등)은
  별도 지표를 더 만드는 대신 **전처리 프로파일**을 바꿔서 처리한다(아래 8번).
- **TEDS-content** (표, 0~1, 높을수록 좋음): 표의 구조 + 셀 내용이 얼마나 일치하는지.
  **docx/pptx 표와 엑셀 모두 이 지표를 쓴다.** 엑셀은 null 패딩 등을 걷어내는
  전용 전처리를 먼저 거친다(아래 9번).

GT 확장자에 따라 갈린다: **docx/pptx → WER + TEDS**, **xlsx → TEDS**(본문 WER은 없음).

---

## 1. 파일 구성

| 파일 | 역할 |
|---|---|
| `config.py` | 경로/옵션 설정 (여기 `DATA_DIR`만 바꾸면 됨) |
| `gt_extract.py` | GT(.docx/.pptx/.xlsx)에서 본문 텍스트 + 표 추출 (서식 복원 포함) |
| `parsed_extract.py` | 파서 결과(txt)에서 본문 텍스트와 표(`<table>`)를 분리 |
| `normalize.py` | WER 계산 전 전처리(정규화) |
| `metrics.py` | WER · TEDS-content 계산 + 진단 도구 |
| `report_html.py` | (선택) GT/Before/After 3-way 비교 HTML 리포트 |
| `run_eval.py` | **단일 실행 진입점** (전체 평가 `run()` + 문서 진단 `diagnose()`) |
| `run_eval.ipynb` | 주피터 실행용 노트북 |

역할별로 모듈을 나눴다: 입력 추출(gt/parsed) · 전처리(normalize) · 지표(metrics) ·
리포트(report_html) · 실행(run_eval). 지표 계산과 실행이 섞이지 않아, 지표를 바꾸거나
새 입력 형식을 추가할 때 해당 파일만 손대면 된다.

## 2. 데이터 준비 (폴더/파일명 규칙)

`config.DATA_DIR`(기본 `./data`) **폴더 하나**에 gt/before/after 파일을 전부 넣는다.
파일명으로 문서와 역할을 구분한다.

```
data/
├── 1-1_gt.docx        ← 정답(원본). 확장자 .docx / .pptx / .xlsx
├── 1-1_before.txt     ← 가이드라인 적용 "전" 파서 결과
├── 1-1_after.txt      ← 가이드라인 적용 "후" 파서 결과
├── 1-2_gt.pptx
├── 1-2_after.txt      ← after만 있어도 됨 (after만 비교)
├── 1-3_gt.xlsx
└── 1-3_before.txt     ← before만 있어도 됨 (before만 비교)
```

파일명 형식:
```
{문서id}_gt.(docx|pptx|xlsx)
{문서id}_before.txt
{문서id}_after.txt
```

- `{문서id}`는 자유(예: `1-1`, `doc_0001`, `card_fee`). 같은 문서의 gt/before/after는
  이 id가 같아야 매칭된다.
- **gt는 필수**(정답이 없으면 비교 불가). **before/after는 둘 중 하나만 있어도 된다.**
  - `1-1_gt` + `1-1_before` + `1-1_after` → before·after 둘 다 비교
  - `1-2_gt` + `1-2_after`               → after만 비교 (before는 결과에서 빈 값)
  - `1-3_gt` + `1-3_before`              → before만 비교
- before/after txt 안에는 원문 순서대로 텍스트(`#`, `**`, `~~`, `-` 등 Markdown 포함)와
  표(`<table>...</table>` HTML)가 섞여 있다고 가정한다.

## 3. 설치 & 실행

설치:
```bash
pip install -r requirements.txt
```

### 주피터에서 실행 (권장)
`run_eval.ipynb`를 열어 위에서부터 셀을 실행한다. 핵심은 두 줄이다.
```python
import run_eval
df, summary = run_eval.run()      # 전체 평가 + 요약 + CSV 저장
```
특정 문서를 자세히 뜯어보려면:
```python
run_eval.diagnose("1-1", stage="before", make_html=True)
```

주피터에서 `.py`를 직접 돌리고 싶으면:
```python
%run run_eval.py
```
(주의: 노트북과 모든 `.py` 모듈, `data/` 폴더가 **같은 작업 폴더**에 있어야 한다.)

### 터미널에서 실행
```bash
python run_eval.py
```

### 바로 테스트해보기 (가상 데이터)
```bash
python make_sample_data.py   # data/ 폴더에 sample_xlsx_*, sample_ppt_* 생성
python run_eval.py           # 샘플로 지표 확인
```
(엑셀은 패딩/`Unnamed` 섞인 파서 txt, PPT는 2번 슬라이드가 누락된 before를 넣어둠.)

### 단위별로 쪼개 보기 (PPT 슬라이드 / 엑셀 시트)
```python
run_eval.by_unit("sample_ppt", stage="before")   # 슬라이드별 WER (## 섹션과 정렬)
run_eval.by_unit("sample_xlsx", stage="before")  # 시트별 TEDS
```

실행하면 콘솔에 문서별 상세 + 전체 요약이 나오고,
`output/report_per_document.csv`(문서별), `output/report_summary.csv`(요약)가 저장된다.

## 4. WER 계산 전 전처리 (normalize.py)

WER은 "토큰이 정확히 같은가"만 보므로, 내용은 같은데 표기만 다른 차이(전각/반각,
따옴표 모양, 공백 개수 등)까지 오류로 센다. 이를 막기 위해 GT/파서 텍스트에 **똑같이**
전처리를 적용한다.

**공정 비교 원칙**: 서식(제목/볼드/취소선/리스트)은 GT와 파서가 같은 표기일 때만 공정하다.
그래서 각 서식은 둘 중 하나로만 처리한다.
- **채점**: GT 추출기가 그 서식을 Markdown 기호로 복원하고, 전처리에서 기호를 지우지
  않고 통일만 한다 → 파서가 서식을 살렸는지/놓쳤는지가 WER에 반영됨.
- **무시**: GT/파서 양쪽에서 기호를 제거한다 → 점수에 영향 없음.

| 항목 | 기본값 | 처리 |
|---|---|---|
| 유니코드 정규화(NFKC) | ON | 전각/반각·자모 조합 통일 (`％`→`%`) |
| 제로폭/제어문자 제거 | ON | 눈에 안 보이지만 토큰을 어긋나게 하는 문자(ZWSP, BOM 등) 제거 |
| 공백/줄바꿈 정규화 | ON | 탭/줄바꿈/연속 공백 → 공백 1칸 |
| 제목 `#` | 유지(채점) | GT도 `#`로 복원 → 제목 보존 여부 채점 |
| 볼드 `**` | 유지(채점) | GT도 `**`로 복원 → 강조 보존 여부 채점 |
| 취소선 `~~` | 유지(채점) | GT도 `~~`로 복원 → 취소선 보존 여부 채점 |
| 리스트 불릿 | 유지·통일(채점) | `-`,`*`,`•`,`·` 등을 `- `로 통일 (GT도 `- `로 복원) |
| 인용 `>` / 구분선 `---` / 링크 / 인라인 코드 | 제거(무시) | GT엔 없는 스타일이라 제거 |
| 대시(-)만 있는 줄 | 제거(무시) | PPT 파서가 슬라이드 경계를 `-` 한 줄로 넣는 경우 대비 (`- 내용` 목록은 보존) |
| 따옴표/대시 통일 | ON | `“ ”`→`"`, `– —`→`-` |
| 영문 대소문자 | 소문자화 | 약어 대소문자 차이 무시 |
| 문장부호 제거 | **OFF** | `1,200,000원`의 콤마까지 지워 실제 오류를 가릴 수 있어 기본 꺼둠 |

각 항목은 `normalize.normalize_for_wer(...)` 인자로 개별 on/off 할 수 있다.
취소선을 "삭제된 글"로 보고 내용까지 빼고 싶으면 `drop_strike_content=True`를 쓴다.

## 5. GT 서식 복원 (gt_extract.py)

GT 원본에는 Markdown 기호가 없고 Word/PPT의 서식 속성으로만 구조가 들어있다.
파서 출력과 같은 기호 체계로 맞추기 위해 GT에서도 아래처럼 복원한다.

- 제목 스타일(Heading N / 제목 N) → `#` * N
- 굵게(bold) run → `**텍스트**`
- 취소선(strike) run → `~~텍스트~~`  (볼드와 겹치면 `**~~텍스트~~**`)
- 목록 스타일 문단 / PPT 들여쓰기 문단 → `- 텍스트`
- 표 → `<table>` HTML (셀 텍스트 기준)

형식별 특징:
- **docx**: 문단/표를 순서대로 순회. 문단 run 단위로 볼드·취소선 판정.
- **pptx**: 슬라이드 → 도형(텍스트 프레임/표) 순회. 문단 들여쓰기 레벨을 목록으로 간주.
- **xlsx**: 시트 하나를 표 하나로 변환(본문 텍스트는 거의 없음). 바깥 빈 행/열은 잘라냄.

## 6. WER의 "단어" 단위 (한국어)

- **어절(eojeol, 기본값)**: 공백으로 자름. 설치 불필요. 한국어 STT/OCR 평가의 표준.
  "아닙니다"/"아니에요"를 다른 토큰으로 봐 활용형 차이도 오류로 센다.
- **형태소(morpheme)**: 형태소 분석기(kiwipiepy)로 어간/어미/조사를 쪼갠 뒤 비교.
  활용형 차이에 관대. `pip install kiwipiepy` 필요.

`config.WER_TOKENIZER`를 `"eojeol"` 또는 `"morpheme"`로 지정한다.

## 7. TEDS-content 계산 (metrics.py)

표는 `table>tr>td`로 얕고 규칙적인 구조라, 범용 트리편집거리(외부 라이브러리) 대신
**표 전용 2단계 편집거리**로 직접 계산한다(동작이 결정적이고 의존성이 없다).
먼저 GT 행들과 파서 행들을 편집거리로 정렬(행 추가/삭제/치환)하고, 짝지어진 두 행
안에서 셀을 다시 편집거리로 정렬한다(셀 치환 비용 = 1 − 텍스트 유사도).
`TEDS-content = 1 − 편집거리 / max(GT·파서 노드 수)`로 계산한다(0~1, 1이면 완전 동일).
빈 셀은 파서가 `null`/`Unnamed: 1`처럼 문자열로 내보내도 GT의 빈 칸과 같게 취급한다
(`metrics._normalize_cell`에서 통일).
GT/파서의 표 개수가 다르면 등장 순서대로 짝지어지는 개수까지만 평균 낸다.

파서가 표를 `<table>` HTML로 내든 마크다운 파이프 표(`| a | b |`)로 내든
`parsed_extract.py`가 둘 다 인식해 내부적으로 같은 표로 변환한다.

## 8. 전처리 프로파일 — 상황별 대응 (config.py)

텍스트 지표는 WER 하나로 유지하되, 문제 상황은 **전처리 케이스(프로파일)**를 바꿔
해결한다. `config.EVAL_PROFILE` 한 줄만 바꾸면 그 케이스로 WER을 계산한다.

| 프로파일 | 토큰 | 용도 |
|---|---|---|
| `default` | 어절 | 기본. 제목/볼드/취소선/리스트 채점 + 표기 통일 |
| `spacing_robust` | 형태소 | **띄어쓰기/줄바꿈 밀림 대응.** 형태소로 재분절해 띄어쓰기 차이를 흡수 (kiwipiepy 필요) |
| `content_only` | 어절 | 서식은 채점 안 하고 순수 내용만 (헤더/볼드/취소선 기호 제거) |

**왜 띄어쓰기 밀림은 형태소가 답인가**: 어절 WER은 공백으로 토큰을 나누므로
"전월 실적"↔"전월실적"이 서로 다른 토큰이 돼 WER이 폭발한다. 공백을 다 지우면
어절 토큰이 1개가 돼 WER 자체가 깨진다. 형태소 분석기는 공백과 무관하게
같은 형태소로 재분절하므로, WER 하나를 유지하면서 밀림을 흡수할 수 있다.
(실측: 띄어쓰기만 다른 문서에서 default WER 0.86 → spacing_robust WER 0.00.)

특정 패턴만 교정하려면 프로파일의 `normalize`에
`custom_regex_subs=[(정규식, 치환)]`를 넣는다(예: 숫자 사이 공백 제거).

어느 케이스가 맞는지 볼 때는 한 문서를 여러 프로파일로 비교한다:
```python
run_eval.compare_profiles("2-1", stage="before")
#   default        (토큰 eojeol  ): WER 0.8571
#   spacing_robust (토큰 morpheme): WER 0.0000   ← 밀림이 원인이었음
#   content_only   (토큰 eojeol  ): WER 0.8571
```

## 9. 엑셀 GT 전용 처리 (xlsx_eval.py)

엑셀은 문서 전체가 표이고, 파서는 시트 전체를 훑어 빈 칸을 `null`로 채운 마크다운
파이프 표로 내보낸다(빈 헤더는 `Unnamed: 1`). 그래서 실제 표 밖 패딩 셀이 잔뜩 붙어
원본과 셀 개수가 무의미하게 달라진다. 이 상태로 바로 TEDS를 내면 패딩 때문에 점수가
왜곡되므로, **TEDS 계산 전에 엑셀 전용 전처리**를 한다.

  1. 셀 표기 통일: `null` / `Unnamed: N` / `nan` 등을 빈 칸으로
  2. 여러 표(파서의 md 표 블록, GT의 여러 시트)를 세로로 이어붙여 한 표로
  3. 완전히 빈 행/열을 (가장자리뿐 아니라 중간까지) 모두 제거 → 패딩 제거
  4. 정리된 GT/파서 표끼리 **TEDS-content** 계산 (docx/pptx와 같은 지표)

즉 엑셀도 결국 표 지표는 TEDS 하나로 통일된다. GT 확장자가 `.xlsx`면 `run_eval.run()`이
자동으로 이 전처리 경로를 탄다. `run_eval.diagnose(doc_id, "before")`로 정리된 표
(패딩 제거 후 실제 비교되는 표)를 눈으로 확인할 수 있다.

> 참고: 표를 하나로 합쳐 비교하므로 파서가 표를 몇 조각으로 나눴는지와 무관하다.
> 다만 정리 후에도 행이 아주 많으면 TEDS(2단계 편집거리) 계산이 느려질 수 있다.

## 10. PPT 슬라이드 / 엑셀 시트 단위 비교 (slide_eval.py)

파서는 새 슬라이드의 시작을 `## 제목` 헤더로 표시한다. 문서를 통째로 이어 WER을
내면 슬라이드 하나가 빠졌을 때 그 뒤가 전부 밀린다. 그래서 **PPT는 슬라이드 단위로
정렬한 뒤 WER**을 낸다.

  - GT는 슬라이드별 텍스트, 파서는 `##` 기준 섹션으로 나눈다.
  - 두 리스트를 슬라이드 단위로 정렬(Needleman-Wunsch)한다. 슬라이드가 빠지거나
    추가돼도 그 슬라이드만 손해 보고 나머지는 제자리에 정렬된다.
  - `run_eval.by_unit(doc_id, "before")`로 슬라이드별 WER을 볼 수 있다
    (누락된 슬라이드는 "파서가 슬라이드 놓침"으로 표시).

엑셀은 시트별 TEDS를 `run_eval.by_unit`으로 확인한다(GT 시트 ↔ 파서 표를 순서로 짝지음).

## 12. 한계

- 표의 병합 셀(rowspan/colspan)은 복원하지 않는다(셀 텍스트 + 기본 행/열 구조까지).
- 중첩 표(표 안의 표)는 다루지 않는다.
- 도형/텍스트 상자/SmartArt 안의 텍스트는 추출하지 않는다.
- 순서 있는 목록의 실제 번호는 복원하지 않고 `- `로 통일한다.
- `<table>` 탐지는 정규식 기반이라 표 HTML이 깨져 있으면(닫는 태그 누락 등) 놓칠 수 있다.
